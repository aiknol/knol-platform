Knol

Long-term memory infrastructure for LLM applications

Built in Rust. Open core. Self-hostable.

Rust + Axum Apache 2.0 11 OSS Crates 4 Microservices PostgreSQL + pgvector NATS JetStream

Overview

Knol gives AI agents persistent, structured memory — not just vector search. Every conversation turn is processed through an intelligent extraction pipeline that builds a knowledge graph of entities, relationships, and facts alongside traditional semantic embeddings. Memories are grounded with source citations, verified for accuracy, and automatically resolved when they conflict.

Why Knol?

Hybrid retrieval — Queries hit vector similarity, knowledge graph traversal, BM25 full-text, and scope cascade simultaneously. The system classifies query intent and routes to the optimal strategy automatically using Reciprocal Rank Fusion (RRF).

Knowledge graph, not just vectors — Every extraction builds entities, relationships, and edges. Ask "who does Sarah work with?" and get graph-traversed answers, not just keyword matches.

Write-time intelligence — Embeddings are generated at write time (not query time). Conflicts are detected automatically. Duplicate memories are deduplicated. Outdated facts are superseded.

Grounded and verifiable — Every extracted memory includes a source citation linking back to the original text. Optional LLM-based verification scores how well the source supports each memory.

Built for production — Rust microservices, NATS JetStream for async event processing, PostgreSQL with pgvector, Redis caching, multi-tenant isolation, JWT auth, and webhook notifications.

Data Flow

Write path: Client → API Gateway → Write Service → NATS JetStream → Graph Service → LLM extraction → Embedding generation → Conflict detection → PostgreSQL + pgvector → Webhook dispatch

Read path: Client → API Gateway → Retrieve Service → Query embedding → Vector search + BM25 + Graph traversal + Scope cascade → RRF fusion → Decay-adjusted results

Key Design Decision

Writes return immediately with an episode ID (sub-50ms). All LLM extraction, embedding, and conflict detection happens asynchronously via NATS JetStream. This decouples write latency from LLM response time.

API Overview

All API endpoints are served through the Gateway at http://localhost:3000. Authentication is via Authorization: Bearer <api-key> header or X-API-Key header.

Authentication

Clients authenticate with an API key that maps to a tenant. The gateway hashes the key with SHA-256 and matches against tenants.api_key_hash. JWT Bearer tokens are also supported.

Gateway Service

Port: 3000 · Crate: service-gateway · Framework: Axum

The gateway is the single public entry point. It handles authentication (API key or JWT), rate limiting (Redis-backed, per-tenant), CORS, tenant context injection, and routes requests to internal services via HTTP.

Responsibilities

  • API key validation (SHA-256 hash lookup in tenants table)
  • JWT token verification with configurable secret
  • Rate limiting per tenant (configurable via system_config)
  • Request routing to write, retrieve, and admin services
  • CORS configuration (configurable origins)
  • Prometheus metrics collection
  • X-Tenant-Id and X-Internal-Secret injection for downstream services

Write Service

Port: 8081 · Crate: service-write · Framework: Axum

The write service provides fast-ACK episode ingestion. It validates input, performs content-hash deduplication, stores the raw episode, and publishes a MemoryWriteEvent to NATS JetStream subject memory.write.* for async processing.

Write Flow

  Client → Gateway → Write Service
                        │
                        ├─ 1. Validate input (max 50KB)
                        ├─ 2. Content hash dedup (SHA-256)
                        ├─ 3. Store episode in PostgreSQL
                        ├─ 4. Publish MemoryWriteEvent to NATS
                        └─ 5. Return 200 OK { episode_id, status: "accepted" }
                           (sub-50ms latency)

Retrieve Service

Port: 8082 · Crate: service-retrieve · Framework: Axum

The retrieve service implements adaptive hybrid search. It classifies query intent using local heuristics (no LLM call), then runs multiple search strategies in parallel, fusing results with Reciprocal Rank Fusion (RRF).

Intent Classification

IntentTrigger KeywordsPrimary Strategy
Preferenceprefer, like, favorite, wantVector-first (weight 1.0)
Temporalwhen, timeline, recently, pastBM25 heavy (0.8) + Graph (0.7)
Relationalwho, relationship, connected, works atGraph-first (weight 1.0)
GeneralDefaultBalanced hybrid (all methods)

4-Stage Fusion Pipeline

1
Vector Search
Real embeddings via configurable provider (OpenAI, Anthropic, Gemini). Cosine similarity on pgvector HNSW index.
2
BM25 Full-Text Search
PostgreSQL ts_rank_cd with optional custom weights [D,C,B,A].
3
Graph Expansion
Entity-based N-hop traversal with configurable max depth (default 10). Follows relationship edges to find connected memories.
4
Scope Cascade
Multi-level scoping: session → user → agent → org. Broadens scope if insufficient results at narrow scope.

RRF Fusion Formula

score(memory) = Σ weight_i / (k + rank_i(memory))

Where:
  k     = RRF constant (default 60.0, configurable)
  w_i   = per-intent weight for each retrieval method
  rank_i = rank of memory in the i-th retrieval method's results

DynamicLlmProvider

The DynamicLlmProvider wrapper re-reads configuration and API keys from the database every 60 seconds. You can switch LLM providers or update API keys via the admin panel with zero service restarts.

Content Triage

Classifies content before LLM calls into: Skip (greetings, acks, <3 words), Light (simple messages), or Full (rich content requiring full extraction). Eliminates 30-60% of unnecessary LLM calls.

Guardrails

  • PII Detection — Regex-based detection of emails, phones, SSNs, credit cards
  • PII Modes — redact, mask, hash, allow
  • Prompt Injection — Heuristic detection before LLM calls

  • Automatic migration execution at startup
  • Row-Level Security (RLS) via SET app.tenant_id per transaction
  • CRUD for episodes, memories, entities, edges

memory-cache

Redis wrapper built on the Fred client library. Provides typed get/set operations with TTL management and specialized helpers for LLM response caching (keyed by content hash).

memory-queue

NATS JetStream producer and consumer. Handles stream creation, durable consumer setup, message serialization, and acknowledgment (ACK on success, NACK for retry). Messages are processed in configurable batches (default 10 concurrent).

Authentication & Security

Tenant Auth (Public API)

Clients authenticate via X-API-Key or Authorization: Bearer header. The gateway hashes keys with SHA-256 and matches against tenants.api_key_hash. JWT Bearer tokens are also supported.

Admin Auth

Admin users log in with email/password (bcrypt). JWT sessions (~8h expiry) stored in admin_sessions table. Roles: super_admin, config_admin, marketing_admin, read_only.

Credential Encryption

LLM API keys and OAuth tokens are stored as AES-256-GCM encrypted bytes. The encryption key comes from ADMIN_ENCRYPTION_KEY env var (base64-encoded, 32 bytes). A random 12-byte nonce is generated per encryption.

Security Features

  • Row-Level Security (RLS) — Every query scoped to current_setting('app.tenant_id')
  • PII Detection/Redaction — Regex-based detection of emails, phones, SSNs, credit cards. Modes: redact, mask, hash, allow
  • Prompt Injection Detection — Heuristic detection before LLM calls
  • Rate Limiting — Per-tenant Redis-backed rate limiter in gateway
  • Internal Secret — Constant-time comparison for inter-service auth (INTERNAL_SERVICE_SECRET)
  • Full Audit Trail — memory_audit, admin_audit_log, llm_usage_log

Key Performance Characteristics

  • Write latency: Sub-50ms (async extraction via NATS)
  • Search latency: ~45ms average (parallel vector + BM25 + graph)
  • Extraction latency: ~800ms average with all optimizations
  • Content deduplication: SHA-256 hash prevents duplicate episodes
  • LLM cache hit rate: Typically 20-40% (configurable TTL)
  • Content triage skip rate: 30-60% of messages classified as trivial
